    ### File structure (verbatim headings)

    ```markdown
    # Refactor Playbook

    This document is the single source of truth for how refactor PRs flow through this
    repo. Every structural change to `src/server.ts`, `src/client/**`, `src/map/**`,
    `src/*/controller.ts`, or any module under refactor must follow it.

    ## Pre-flight Checklist

    Before editing any code, run this sequence. Skipping a step has caused real production regressions in this codebase; the checklist exists to make that not happen again.

    1. **Index freshness.** `npx gitnexus analyze` (or `--embeddings` if `.gitnexus/meta.json` shows `stats.embeddings > 0`) — required if any commit landed since your last local analyze.
    2. **Impact analysis.** `gitnexus_impact({target: "<symbol-you-are-moving>", direction: "upstream"})`. Read the d=1 list carefully. If ANY result is labeled HIGH or CRITICAL, stop and reassess before proceeding.
    3. **Codebase state clean.** `git status` shows no unrelated work in flight; branch is off the latest `main`.
    4. **Harness baseline present.** `.fixtures/mud-traffic-baseline.log` exists locally (re-extract if older than ~1 week — see "Baseline Fixture Restoration"). `snapshots/parser-before.jsonl` and `snapshots/replay-before.jsonl` are checked in at HEAD.
    5. **Green before changes.**
       - `bun run typecheck` passes
       - `bun test` passes
       - `bun run parser:snapshot` exits 0 ("zero diff")
       - `bun run replay:check` exits 0 ("zero diff")

       If any of these fail BEFORE your changes, the main branch is broken — do not start until it's fixed.

    ## Commit Convention

    - PR title: `refactor(N): <what>` where N is the phase number and `<what>` is a short imperative description.
    - Examples:
      - `refactor(2): extract stats-parser controller`
      - `refactor(2): subscribe loot-sort to bus via shim`
      - `refactor(3): split map-grid into layout + render + interactions`
    - One PR = one structural extraction + the minimum wiring needed to keep behaviour-preserving invariant.
    - NO mixed-purpose PRs. If you notice a bug during extraction, file it as a separate issue and leave the bug intact — do not "fix it while I'm in there" (see PITFALLS.md Pitfall 5, antipattern A1).
    - Every commit in a refactor PR ends with the regression gate passing (checklist below).

    ## Regression Definition

    Behaviour-preserving means **strict byte-equality**:

    - `bun run parser:snapshot` — `snapshots/parser-after.jsonl` byte-identical to `snapshots/parser-before.jsonl`.
    - `bun run replay:check` — `snapshots/replay-after.jsonl` byte-identical to `snapshots/replay-before.jsonl`.

    There is no tolerance margin. A single whitespace change in a rendered event payload is a regression.

    Exception — intentional surface growth: Phase 2 extractions that wire new bus subscribers WILL change the replay transcript (new `mapStore.*`, `timer.*`, or `mudCommandSink.*` entries appear). When this is the intent:
    1. Review the diff manually.
    2. Confirm every new entry represents a side-effect that ALSO existed in the pre-extraction code path (same DB writes, same outgoing commands, same broadcasts).
    3. Re-run `bun run replay:check --write-initial` to re-seed `snapshots/replay-before.jsonl`.
    4. Commit the re-seeded file IN THE SAME PR as the extraction, with a commit body documenting each new `kind` added.

    Phase 4 TEST-* PRs may similarly re-seed snapshots if they wire new test-side observers.

    ## Baseline Fixture Restoration

    `.fixtures/mud-traffic-baseline.log` is gitignored (CONTEXT D-04). Regenerate locally:

    ```bash
    bun run scripts/extract-baseline.ts --start <ISO-8601 timestamp from the live log>
    ```

    How to pick a good start timestamp:
    - Inspect `/var/log/bylins-bot/mud-traffic.log` for a 30-minute window with diverse activity (farm + chat + bazaar + repair + survival — CONTEXT D-02).
    - Good heuristic: grab a window where the bot was actively farming (`grep direction=mud-out /var/log/bylins-bot/mud-traffic.log | head -100` to find recent farming sessions; pick the ISO timestamp of a command within that range).
    - Default window is 30 minutes (`--minutes 30`); the extraction script exits code 2 if no lines match, code 1 on fs errors.

    The fixture is NEVER committed. Only the scripts that read it (`parser-snapshot`, `replay-harness`) and the committed snapshots (`snapshots/parser-before.jsonl`, `snapshots/replay-before.jsonl`) are in git.

    If you're onboarding and need the exact baseline another developer used, coordinate via out-of-band channel (your pair partner / prior commit author).

    ## Running Harnesses Locally

    ```bash
    # First time after extract-baseline.ts has produced the fixture:
    bun run parser:snapshot --write-initial    # seeds snapshots/parser-before.jsonl (one-off)
    bun run replay:check --write-initial       # seeds snapshots/replay-before.jsonl (one-off)

    ## Installing the Pre-commit Hook

    The hook lives at `.githooks/pre-commit`. Git does not use `.githooks/` by default — you opt in on each local clone:

    ```bash
    git config core.hooksPath .githooks
    chmod +x .githooks/pre-commit   # only if the file isn't already executable
    ```

    What it does:
    - If your current branch matches `refactor/*` OR the commit message you are typing contains `refactor(`, the hook runs `bun run replay:check`.
    - If the harness reports REGRESSION (exit 1), the commit is blocked.
    - If the harness exits 2 (fixture missing), the hook surfaces the error — re-run `bun run scripts/extract-baseline.ts --start <ISO>` to regenerate the fixture.
    - On non-refactor branches/commits, the hook is a no-op (fast exit).

    CI note: there is no CI pipeline in this milestone (v2 TOOL-01 defers). The pre-commit hook is the only automated regression gate; discipline is the other half.

    If you need to bypass the hook in an emergency: `git commit --no-verify`. Use this only if the harness is broken (infrastructure issue), never to skip a genuine regression. If you `--no-verify` a real regression into main, you own the rollback.

    ### File contents (exact — verbatim)

    ```bash
    #!/usr/bin/env bash
    # Pre-commit hook: runs the replay-harness on refactor branches/commits.
    # Enabled per-developer via: git config core.hooksPath .githooks
    # Bypass in emergency (only for broken-harness cases): git commit --no-verify

    set -euo pipefail

    branch="$(git symbolic-ref --short HEAD 2>/dev/null || echo '')"
    commit_msg_file="${1:-.git/COMMIT_EDITMSG}"
    commit_msg=""
    if [[ -f "${commit_msg_file}" ]]; then
      commit_msg="$(cat "${commit_msg_file}" 2>/dev/null || echo '')"
    fi

    is_refactor=0
    if [[ "${branch}" == refactor/* ]]; then
      is_refactor=1
    elif [[ "${commit_msg}" == *"refactor("* ]]; then
      is_refactor=1
    fi

    if [[ ${is_refactor} -eq 0 ]]; then
      exit 0
    fi

    echo "[pre-commit] refactor context detected (branch='${branch}'); running replay:check"
    if ! bun run replay:check; then
      echo "[pre-commit] replay:check failed — refusing commit. Fix the regression or run with --no-verify if the harness itself is broken." >&2
      exit 1
    fi

    exit 0
    ```
