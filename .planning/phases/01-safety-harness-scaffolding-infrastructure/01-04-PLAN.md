<objective>
Adopt a proper Postgres migration framework and drain all inline DDL out of `src/map/store.ts::initialize()`. This is the **highest-risk plan of Phase 1** because the production database has been running with inline `ALTER TABLE IF NOT EXISTS` statements since day one — any naive introduction of `schema_migrations` tracking would double-apply destructive guards (see PITFALLS.md §5). The runner must therefore implement the **baseline-pump** strategy from D-31: on first-run against a production DB (schema_migrations absent, map_rooms present) it SEEDS the tracking table with every known migration ID without executing their SQL; on a fresh install (both tables absent) it applies normally.

Purpose: INFRA-03 + INFRA-04. Unlocks future schema evolution through numbered migrations; removes the "schema drift" trap at `src/map/store.ts:178-293` where inline DDL is invisible to `gitnexus` and unreviewable in PRs.

Output: (1) `src/map/migrations/runner.ts` — the runner + baseline-pump logic; (2) three numbered migration `.sql` files matching the current production DDL; (3) a `bun:test` unit test proving the baseline-pump behaviour; (4) a modified `src/map/store.ts` where `initialize()` is exactly one line: `await runMigrations(database)`; (5) new `migrate` / `migrate:status` scripts in `package.json`.
</objective>

### Baseline-pump state machine (D-31 — CRITICAL):

```
state1 = (schema_migrations EXISTS): apply all unregistered ids normally
state2 = (schema_migrations ABSENT && map_rooms EXISTS): PRODUCTION-UNMIGRATED
  → CREATE TABLE schema_migrations (...)
  → INSERT every known migration id with applied_at = NOW()
  → DO NOT execute the migration SQL for any of them
state3 = (schema_migrations ABSENT && map_rooms ABSENT): FRESH-INSTALL
  → CREATE TABLE schema_migrations (...)
  → apply every migration normally in lexicographic order
```

### Advisory lock strategy (D-33):

Advisory lock id `727465` (decimal hash "bylins" — literal constant). Use `pg_advisory_xact_lock()` (transaction-scoped, auto-released on tx end), NOT session-scoped, to avoid leak on process crash.

```typescript
await database.begin(async (tx) => {
  await tx`SELECT pg_advisory_xact_lock(727465)`;
  // ... all reads + writes happen inside this block
});
```

### Error handling (D-35):

Fail-fast. Any error inside `database.begin(...)` aborts the transaction (postgres.js rolls back automatically on throw) and re-throws. `runMigrations` does not catch — caller decides. `mapStore.initialize()` does not catch either — startup failure of migrations MUST crash `bun run start`. This matches the existing failure semantic for `DATABASE_URL missing` (`src/config.ts:110` — throw-to-refuse-start).

Narrowing pattern for enriching error messages (CONVENTIONS.md:72):
```typescript
try {
  await applyMigration(tx, id, sqlText);
} catch (error: unknown) {
  throw new Error(`[migrations] failed applying ${id}: ${error instanceof Error ? error.message : "unknown"}`);
}
```

### Migration filename convention (D-32):

`YYYYMMDDHHMMSS-short-kebab-description.sql` — lexicographic = chronological. All three Phase 1 migrations timestamped `20260418180000`, `20260418180100`, `20260418180200` (fixed values — do not pick "now" timestamps; the IDs are load-bearing and referenced in the SEED step of baseline-pump).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| developer → migration files | Migration SQL shipped in the repo. Developer controls contents; reviewed via PR. No runtime user input reaches SQL. |
| startup → production DB | `runMigrations(database)` runs on every `bun run start` — touches production schema during bootstrap. Single-writer invariant enforced by advisory lock. |
| CI/CLI → `bun run migrate` | Same code path as startup, invoked explicitly for dry-runs. Inherits all runtime trust boundaries; does not introduce new ones. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-04-01 | Tampering | Migration SQL files | accept | Files shipped in repo; tampering = merge compromise. Out of scope for Phase 1 threat model; covered by GitHub branch protection (v2 TOOL-01). |
| T-01-04-02 | Tampering | Runtime SQL injection via migration path | mitigate | Zero user input flows into SQL. Runner uses `sql.unsafe()` on committed files only; filename lookup is `readdir(__dirname)` + regex filter on `/^\d{14}-[a-z0-9-]+\.sql$/`. No filename comes from env, args, or DB. Acceptance: `grep -cE "process\.env\|Bun\.env" src/map/migrations/runner.ts` returns 0. |
| T-01-04-03 | Repudiation | Migration application not traceable | mitigate | Every applied migration produces an `[migrations] applied <id>` line via the injected `deps.onLog` (D-35). Runner also inserts `(id, applied_at = NOW())` into schema_migrations, preserving an on-DB audit trail independent of logs. Mitigation verified by test case "normal run logs each applied id". |
| T-01-04-04 | Information Disclosure | Baseline SQL leaks secrets | mitigate | Baseline.sql is captured via `pg_dump --schema-only --no-owner --no-privileges` — **schema only**, no data, no roles. Acceptance (enforced in Task 2): `grep -iE "password\|secret\|token\|respect1\|api[_-]?key" src/map/migrations/20260418180000-baseline.sql` returns 0 matches. |
| T-01-04-05 | Denial of Service | Concurrent migration runners deadlock or corrupt | mitigate | D-33 transaction-scoped advisory lock `pg_advisory_xact_lock(727465)` serializes migration work. Second runner blocks until first commits, then sees an up-to-date `schema_migrations` and applies zero migrations. Even with future multi-instance PM2 (currently single-instance), cheap insurance. |
| T-01-04-06 | Denial of Service | Baseline-pump misfires and re-runs destructive migration on production | mitigate | **This is the #1 pitfall from PITFALLS.md §5.** Mitigation: runner state machine detects "production-unmigrated" via `(schema_migrations absent) AND (map_rooms present)` BEFORE executing any SQL; in that branch it seeds schema_migrations and exits without running any migration file. Unit test (Task 4, case 2) proves this path never calls `tx.unsafe()`. |
| T-01-04-07 | Elevation of Privilege | Migration runs with more privilege than intended | accept | The runner uses the existing `DATABASE_URL` connection, which has the same privileges as the running bot (schema owner). No privilege escalation is introduced; fixing role separation is out of scope (v2 SEC-*). |
| T-01-04-08 | Spoofing | N/A | accept | No authentication boundary inside the runner. Connection is trusted at the `DATABASE_URL` level; orthogonal to this plan. |
| T-01-04-09 | Tampering | Failed migration leaves DB half-applied | mitigate | D-35 fail-fast: every migration runs inside `database.begin(...)` — postgres.js rolls back the transaction on throw, schema_migrations row for that id is NOT inserted, next deploy re-runs cleanly. Acceptance: unit test "mid-migration error rolls back" (Task 4, case 4). |

    ```typescript
    export async function runMigrations(
      database: DatabaseClient,
      deps: MigrationRunnerDependencies = NOOP_DEPS,
    ): Promise<void> {
      const files = await listMigrationFiles();

      await database.begin(async (tx) => {
        await tx`SELECT pg_advisory_xact_lock(${ADVISORY_LOCK_ID})`;

        await tx`
          CREATE TABLE IF NOT EXISTS schema_migrations (
            id TEXT PRIMARY KEY,
            applied_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
          )
        `;

        const appliedRows = await tx<{ id: string }[]>`SELECT id FROM schema_migrations`;
        const appliedIds = new Set(appliedRows.map((r) => r.id));

        if (appliedIds.size === 0) {
          const roomsExistsRows = await tx<{ exists: boolean }[]>`
            SELECT EXISTS (
              SELECT 1 FROM information_schema.tables
              WHERE table_schema = 'public' AND table_name = 'map_rooms'
            ) AS exists
          `;
          const mapRoomsExists = roomsExistsRows[0]?.exists === true;

          if (mapRoomsExists) {
            // PRODUCTION-UNMIGRATED path: seed every known migration id as applied; do NOT execute.
            // Sequential by design — migrations apply in order; Promise.all would race.
            for (const filename of files) {
              const id = filename.replace(/\.sql$/, "");
              await tx`INSERT INTO schema_migrations (id) VALUES (${id})`;
              deps.onLog(`[migrations] baseline-pump seeded ${id} (not executed)`);
              appliedIds.add(id);
            }
            return;
          }
          // else FRESH-INSTALL: fall through to NORMAL apply loop
        }

        // NORMAL path: apply each unregistered migration in lexicographic order.
        // Sequential by design — migrations apply in order; Promise.all would race.
        for (const filename of files) {
          const id = filename.replace(/\.sql$/, "");
          if (appliedIds.has(id)) {
            continue;
          }
          const sqlText = await readFile(join(MIGRATIONS_DIR, filename), "utf8");
          try {
            await tx.unsafe(sqlText);
          } catch (error: unknown) {
            throw new Error(`[migrations] failed applying ${id}: ${error instanceof Error ? error.message : "unknown"}`);
          }
          await tx`INSERT INTO schema_migrations (id) VALUES (${id})`;
          deps.onLog(`[migrations] applied ${id}`);
        }
      });
    }
    ```
